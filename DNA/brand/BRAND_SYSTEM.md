## 1. BRAND IDENTITY

### The Funnel Model (Awareness → Ownership)

This document governs the Ownership Layer of the Evolution Funnel.  
Evolution Intelligence (Awareness Layer) earns attention. Evolution Stables
(Ownership Layer) converts qualified, self-selecting participants into regulated
digital-syndication owners.

Internal model:

- Awareness Layer: Fast, accessible, democratic, high-energy. Governed by
  `INTELLIGENCE_SYSTEM.md`.

- Ownership Layer: Mature, refined, long-horizon, and stewardship-led.
  Governed by this file.

### The Progression

Stage 1 — Awareness  
Evolution Intelligence builds trust through data accuracy and pundit auditing.
User follows for racing intelligence. No ownership content. No Evolution Stables presence.

Stage 2 — Discovery  
User discovers Evolution Stables through algorithmic suggestion, search ("own a racehorse NZ"),
or word of mouth from existing participants. First touch: platform explainers, regulatory
credentials, DRC partnership validation.

Stage 3 — Education  
User explores the digital-syndication model, FMA/VARA compliance, Tokinvest infrastructure,
and secondary marketplace mechanics. Content: Bloomberg-style explainers, "How It Works" documentation.

Stage 4 — Conversion  
User becomes a digital-syndication participant. Onboarding through MyStable dashboard,
Evo Studio, legal documentation.

Critical: The progression is always user-initiated. Awareness-layer content never pushes
directly toward Evolution Stables. The wall between Awareness and Ownership is permanent
and non-negotiable.

### The Promise

"Ownership, evolved."

Two words. Not a tagline. A position. Racing ownership has been opaque, illiquid, and
relationship-gated for centuries. We are the regulated, transparent, accessible alternative.
We do not announce this. We demonstrate it through infrastructure, compliance, and the
quality of what we build.

### What We Are
- FMA-Regulated Platform (New Zealand Equine Exemptions)
- Blockchain tokenisation of fractional racehorse ownership
- Transparent marketplace for trading ownership stakes
- Institutional-grade investment opportunity

### What We Are Not
- Traditional relationship-driven syndicate
- Gambling platform
- Entertainment-first racing club
- Amateur horse ownership scheme

### Non-Negotiable Values
- Clarity over complexity
- Confidence without noise
- Regulation first
- Human before technical
- Progress with purpose

### The Fight Club Rule
We do not sell ownership. We sell belonging to a sophisticated moneyball group.
The ownership product is the reward. It is never the headline.

## 2. REGULATORY ARCHITECTURE

### Two-State Ownership Model

Evolution Stables operates in two distinct regulatory states.  
These are NOT interchangeable. Content, claims, and Tokinvest references differ between states.

#### NZ Ownership — Live

Regulator:     NZTR (NZ Thoroughbred Racing) — Authorised Syndicator  
Compliance:    FMA Equine Exemptions (NZ Financial Markets Authority)  
Market:        New Zealand domestic investors  
Content:       NZTR-regulated claims only  
Tokinvest:     Infrastructure layer — NOT mentioned in NZ-domestic investor communications  
DRC:           Not a primary credential for NZ domestic audience  

#### International Ownership — Pipeline

Regulator:      VARA (Dubai Virtual Assets Regulatory Authority)  
Infrastructure: Tokinvest  
Markets:        Dubai, Hong Kong, UK (in order of rollout)  
Content:        VARA-regulated claims, global digital-syndication framing  
Tokinvest:      "Powered by Tokinvest infrastructure" — appears in:
                - Investor decks (International Ownership)
                - Platform footers (International pages)
                - Regulatory disclosures (International)
                - Press releases (dual Evolution + Tokinvest sign-off)
                NEVER in: Awareness-layer content, NZ domestic investor materials  
DRC:            Primary institutional validation credential  

#### State Separation Rule

Never combine NZTR and VARA claims in the same piece of content.  
If content is NZ domestic: cite NZTR + FMA only.  
If content is international: cite VARA + Tokinvest + DRC where appropriate.  
When in doubt: consult regulatory sign-off checklist before publishing.

### Founder Personal Rules

The following rules apply to all Founder Personal voice output. They are
operational — not aspirational. Every draft must satisfy these before send.

**Posture**
- Default to calm, measured, and operational.
- Lead with structure and facts: what changed, why it matters, what happens next.
- Never oversell. If the value is real, it can be described in one or two plain sentences.

**Returns and Numbers**
- Always use concrete figures (e.g. NZD amounts, lease percentages, token counts).
- Restate that returns are variable and not guaranteed wherever financial context appears.
- Distinguish clearly between distributions, race earnings, and indicative returns.
- Where appropriate, cross-check wording against the current PDS and Syndicate Agreement
  to ensure distributions and return language matches the legal documentation.

**Regulatory Framing**
- Cite regulatory anchors (NZTR, FMA Equine Exemptions, VARA, Tokinvest, DRC) only where contextually appropriate and already permitted in this document.
- Never mix NZTR/FMA and VARA/Tokinvest claims in the same content unless explicitly a cross-border discussion.
- When citing an exemption or code, use the full reference once, then abbreviate.

**Email Close**
- Every outbound that requires a response should close with:
  1. A clear invitation to correct or clarify any misinterpretation.
  2. A forward-looking statement about the next step or continued working relationship.
- Never close with a threat, a point-scoring statement, or an open-ended ultimatum.

**Spelling and Grammar**
- British English throughout: -ise not -ize, -our not -or, behaviour not behavior.
- Active voice always. No nominalisations ("utilisation", "facilitation of").
- One sentence per idea. If a sentence needs a comma, it probably needs splitting.

**What Founder Personal Is Not**
- Not chatty: no filler, jokes, slang, or exclamation marks in Ownership Layer comms.
- Not promotional: no hype language, no superlatives, no implied guarantees.
- Not combative: when correcting others, focus on aligning documents, systems, and outcomes — not on being right.
- Not technical-first: avoid leading with blockchain/crypto jargon. Lead with regulation, investor clarity, and operational benefit.

### Tone Spectrum

Professional but Not Stuffy  
✅ "Here's how our tokenisation works..."  
❌ "One humbly submits that our blockchain infrastructure..."

Confident but Not Arrogant  
✅ "We've built the first FMA-regulated tokenised horse platform"

Technical but Not Jargon-Heavy

### Hard Language Rules

- Tone: Calm, measured, evidence-grounded  
- Format: Plain English (British spelling), declarative sentences  
- Prohibited: No hype, no clichés, no exclamation marks, no Web3 jargon  
- Active voice always

## 4.5 CTA LANGUAGE RULE

The CTA must match the user's actual position in the funnel. Never blur the line
between awareness and transactional language.

### Product CTAs (Marketplace, MyStable, any transactional surface)

Use direct, transactional language that reflects the user's real position:

| User State | CTA | Context |
|---|---|---|
| Onboarded, can purchase | **Buy Now** | Marketplace detail page, MyStable |
| Verified but not yet onboarded | **Finish Verification** | Onboarding flow |
| New user, not verified | **Join MyStable** | Registration/signup surface |

Rules:
- Never use "Apply to Invest" on a product page where the user can directly purchase.
- Never use "Invest Now" — it implies a financial return promise.
- "Join MyStable" aligns with the Fight Club Rule: we sell belonging, not investment.
- Gold accent (`#d4a964`) on primary CTAs. Never more than one gold CTA per surface.

### Awareness CTAs (Landing pages, press, marketing, email campaigns)

Use softer, funnel-appropriate language for top-of-funnel users:

| CTA | Context |
|---|---|
| **Apply to Invest** | Landing pages, awareness campaigns |
| **Submit Interest** | Waitlist or early-access pages |
| **Learn More** | Explainer pages, first touch |

Rules:
- These users are expressing interest, not transacting.
- Never use "Buy Now" on awareness surfaces — it violates the funnel wall.
- Never combine NZTR and VARA claims in the same CTA context.

### The Funnel Wall

The wall between Awareness and Ownership CTAs is permanent and non-negotiable
(per §1). A user on an awareness page sees "Apply to Invest." A user on the
marketplace sees "Buy Now." These must never cross.

## 6. WRITING PATTERNS

### Content Structure

Always: Problem → Solution → Proof
- Problem: Traditional syndicates lock up capital
- Solution: Our tokenised platform enables trading
- Proof: Partnership with Dubai Racing Club validates approach

Never: Feature dump without context  
❌ "We have blockchain! And AI! And a marketplace!"

### Headlines

Formula: [Action/Outcome] + [Differentiator]  
✅ "Trade Racehorse Ownership Like Stocks"  
✅ "New Zealand's First FMA-Regulated Tokenised Horse Platform"  
❌ "We're Revolutionizing Everything!" (hype, no value prop)  
❌ "Syndicates 2.0" (not differentiated enough)

### Technical Explanations — Show, Don't Tell

✅ "When you buy 5% of a horse, you get tokens representing that stake. Trade them anytime."  
❌ "Our proprietary tokenisation infrastructure enables fractional ownership distribution"

### Good vs. Bad Examples

Landing Page Headline  
❌ "Revolutionizing Horse Racing with Blockchain!"  
✅ "Trade Racehorse Ownership Stakes on New Zealand's First FMA-Regulated Platform"

Product Description  
❌ "Our innovative platform leverages cutting-edge blockchain technology to democratize access."  
✅ "Buy fractional ownership in racehorses. Track performance transparently. Sell anytime on our marketplace. All regulated under New Zealand's FMA."

### Founder Email Structure

Pattern: **Context → Agreement → Specifics → Ask → Alignment**

Use this structure for all partner-facing, investor-facing, and regulatory emails
written in the Founder Personal voice.

#### 1. Context
Brief reminder of the prior conversation or decision. One sentence maximum.

> "As per our conversation about moving the HLT issuance model..."

#### 2. Agreement / Shared Understanding
One sentence reflecting mutual alignment.

> "We both agreed that shifting issuance to Evolution Stables while Tokinvest lists the corresponding HLTs would benefit both parties."

#### 3. Specifics
Use numbered points. Cover:
- What is changing (structure, process)
- Why it matters (regulatory, operational, investor clarity)
- Key numbers (percentages, amounts, dates) where relevant

#### 4. Ask
Explicitly state what you need from the recipient.

> "Could you please advise the team on the required website updates..."

#### 5. Alignment / Close
Invite corrections and signal momentum.

> "Please amend or add detail where needed, and I look forward to working together on this more streamlined model."

#### Example (Partner Alignment — Issuance Structure)

Hi [Name],

As per our conversation, moving from the current model where [Party A] issues and lists HLTs to a model where Evolution Stables issues and [Party A] lists the corresponding HLTs has clear merits and would be preferable given the current workflows.

You outlined that this process would be similar to existing models [Party A] is working on (just in reverse), and we both agreed that moving to this structure would benefit both parties.

The two key benefits and changes to the existing workflow under this strategy are:
1. The issuance process for each HLT moves from requiring VARA approval to a notification-based process.
2. Evolution Stables can act as a payment and allocation aggregator, removing the current pain point for non-AED holders.

Could you please advise the team on the required updates to the website and disclosure documentation?

Please amend or add detail where needed, and I look forward to working together on this more streamlined model.

#### Example (Investor Returns — PDS-Aligned)

Distributions are calculated based on 75% of the horse's eligible race earnings and allocated pro-rata to investors according to their token holdings.

For this race:
- Syndicate leasehold interest: 5%, split into 20 tokens of 0.25% each.
- Indicative gross return per token: approximately NZD $37.50.
- Distributions are paid quarterly. Earnings from April–June are scheduled for payment in July.

Returns are variable and not guaranteed. Past performance is not indicative of future results.

**Rule:** Answer first. Explain second — only if asked or if the answer alone is insufficient.

**What to cut:**
- Articles ("the", "a", "an") unless a phrase is incomprehensible without them
- Throat-clearing ("Great question!", "Of course!", "Absolutely!")
- Hedging padding ("I think you might...", "It seems like perhaps...")
- Multi-paragraph preambles. One sentence is often enough.
- Sign-off summaries at the end of already-complete answers.

**Intensity levels:**

| Level | Use when | Result |
|-------|----------|--------|
| **Lite** | Formal or external-facing output | Concise but grammatically complete |
| **Full** | Default. Internal conversations, Q&A, code reviews | ~75% token saving, same accuracy |
| **Ultra** | High-volume contexts, rapid iteration | Maximum telegraphic compression |

**What stays:** Technical accuracy, file paths, code, commands, numbers, dates, version references — anything precise passes through untouched.

**When to expand:**
- Delivering a full implementation (not just an answer)
- Explaining a concept the user demonstrably doesn't know
- When the user explicitly asks for detail

**Example — Full Caveman:**

❌ "Great question! Let me explain what's happening here. When you create a new object reference on each render in React, any component that receives that object as a prop will see it as a brand new reference, which means React will treat it as a changed prop and trigger a re-render even if the underlying data hasn't actually changed."

✅ "New object ref each render. Inline object prop = new ref = re-render. Wrap in `useMemo` or lift state."

**Caveman is not:** Dumber. It is the same brain, less mouth. Technical depth is never reduced — only filler.
